# Google Drive CMS Architecture and Operation Plan

## 1. System Overview

This system is a static site generation CMS that uses Google Drive as a headless CMS backend and Next.js / React as the frontend.

The system removes the need for a custom CMS admin panel. Site operators manage content entirely through:

- Google Docs for writing page and post content
- Google Sheets for publication status, metadata, and design controls
- Google Drive folders for information architecture and routing

The frontend receives normalized content from the backend API and renders it as a static website using Next.js and Tailwind CSS.

## 2. Core Concept

The system separates content structure from visual presentation.

Google Docs is treated as the editing interface for semantic content. Google Sheets is treated as the control plane for publishing, metadata, and design configuration. Next.js combines both sources at build time or revalidation time.

This makes the authoring workflow familiar to non-technical users while keeping the public website clean, fast, and maintainable.

## 3. Data Separation Design

### 3.1 Content Layer: Google Docs

Google Docs is used only for article and page content.

Supported content elements:

- Headings
- Paragraphs
- Lists
- Tables
- Links
- Inline emphasis
- Basic images, if supported in the implementation phase

The system extracts clean semantic HTML from each document. Unnecessary style information generated by Google Docs or Word-derived documents is discarded.

The goal is not to reproduce the document layout exactly. The goal is to convert document structure into clean website content.

### 3.2 Metadata and Design Layer: Google Sheets

Google Sheets manages publication and rendering controls for each document.

Expected columns:

| Column | Purpose |
| --- | --- |
| `title` | Display title |
| `slug` | URL slug |
| `docId` | Google Docs document ID |
| `status` | `draft`, `published`, `archived` |
| `publishedAt` | Publication date and time |
| `category` | Content category |
| `excerpt` | Optional summary |
| `theme` | Design theme class name |
| `accentColor` | CSS variable value for accent color |
| `layout` | Layout variant |
| `updatedAt` | Last detected update time |

The spreadsheet may also define data validation rules such as dropdowns for status, theme, and layout.

### 3.3 Rendering Layer: Next.js and Tailwind CSS

Next.js receives normalized JSON and clean HTML from the backend API.

At render time, the frontend applies:

- Clean HTML extracted from Google Docs
- Metadata from Google Sheets
- Theme class names
- CSS variables such as accent colors
- Layout-specific React components

This enables visual customization without allowing document-level inline styles to leak into the public website.

## 4. Google Drive Directory and Routing Design

Google Drive folder hierarchy maps directly to the public URL structure.

### 4.1 `/Posts`

The `/Posts` folder contains dynamic content such as news, notices, and blog articles.

Publishing rules:

- The folder is linked to a management spreadsheet.
- Only rows with `status = published` are eligible for rendering.
- If `publishedAt` is set, the content is rendered only after that date and time.
- Draft and archived content are excluded from public builds.

Example public routes:

- `/posts/spring-cleaning`
- `/posts/event-notice`
- `/news/maintenance`

The exact route prefix can be configured per project.

### 4.2 `/Pages`

The `/Pages` folder contains fixed pages.

Examples:

- `top.docx` or `top.gdoc` -> `/`
- `about.docx` or `about.gdoc` -> `/about`
- `service.docx` or `service.gdoc` -> `/service`

File names are converted into URL slugs. A reserved file name such as `top` or `index` can be mapped to the root route.

### 4.3 `/Forms`

The `/Forms` folder manages Google Forms or form-related references.

Google Docs content can include a special token:

```text
[Form: https://docs.google.com/forms/...]
```

The parser detects this token and converts it into an embedded iframe component.

The frontend should render forms through a controlled React component rather than directly trusting arbitrary iframe HTML.

## 5. Automatic Setup Flow

The system should reduce setup friction for non-technical operators.

### 5.1 Initial Folder Detection

When a user creates or uploads a document into a managed Google Drive folder, the backend detects the change through a webhook or scheduled sync.

### 5.2 Spreadsheet Auto-Creation

If no management spreadsheet exists in the folder, the backend creates one automatically.

The spreadsheet includes:

- Required columns
- Default dropdown values
- Example row
- Basic usage notes

### 5.3 Row Auto-Registration

When a new document is detected, the system adds a row to the spreadsheet.

Default values:

| Field | Default |
| --- | --- |
| `title` | Document title |
| `slug` | Slug generated from document title or file name |
| `docId` | Google Docs document ID |
| `status` | `draft` |
| `publishedAt` | Current date or empty |
| `theme` | Project default theme |
| `layout` | Project default layout |

The user then changes the status to `published` and selects a design theme from dropdowns.

## 6. Infrastructure Design

### 6.1 Backend API

The backend API can run on Xserver VPS or an equivalent VPS environment.

Responsibilities:

- Google Drive API authentication
- Google Sheets API authentication
- Google Docs export or document retrieval
- HTML parsing and sanitization
- Metadata normalization
- Folder and spreadsheet synchronization
- Optional webhook receiver
- Cache and revalidation orchestration

Possible implementation stack:

- Node.js / TypeScript
- Express, Fastify, Hono, or Next.js API routes
- Google APIs client
- Mammoth for `.docx` parsing if uploaded Word files are supported
- HTML sanitizer such as `sanitize-html`

The backend should expose normalized content endpoints rather than exposing raw Google API responses to the frontend.

Example endpoints:

```text
GET /api/sites/:siteId/posts
GET /api/sites/:siteId/posts/:slug
GET /api/sites/:siteId/pages/:slug
POST /api/webhooks/google-drive
POST /api/sync/:siteId
```

### 6.2 Frontend

The frontend is a Next.js application.

Responsibilities:

- Generate static routes from backend content
- Render pages and posts with React components
- Apply Tailwind CSS themes and layout variants
- Use ISR or on-demand revalidation for updates
- Serve fast static pages to visitors

Deployment options:

- Docker Compose on VPS
- Docker Compose for local development
- Vercel, optional later if a managed frontend deployment is useful
- Static export where dynamic revalidation is not required

### 6.3 Hosting Split

Recommended initial deployment:

```text
Xserver VPS
- Backend API
- Google API integration
- Sync worker
- Cache
- Next.js frontend
- Static page delivery
- Docker Compose runtime
```

The initial production target is a single VPS-hosted Docker Compose deployment. The backend remains the integration hub. The frontend stays lightweight and presentation-focused.

The frontend and backend can be split later if deployment or scaling requirements justify it.

## 7. AI Policy

The system itself does not implement internal AI processing in the initial design.

Instead, customers use Google Workspace Gemini or Google AI Pro at the document creation stage for:

- Drafting
- Proofreading
- Rewriting
- Summarization
- Tone adjustment

This avoids direct LLM API costs and reduces system maintenance burden.

The CMS remains focused on:

- Fetching content
- Cleaning structure
- Applying design
- Publishing the site

## 8. MVP Scope

The first implementation should be intentionally small.

### 8.1 Must Have

- Google Drive folder configuration
- Google Sheets-based post index
- Google Docs content retrieval
- Clean HTML extraction
- Published/draft filtering
- `publishedAt` filtering
- Next.js post list page
- Next.js post detail page
- Fixed page routing from `/Pages`
- Basic Tailwind theme mapping

### 8.2 Should Have

- Spreadsheet auto-creation
- New document auto-registration
- Manual sync button or endpoint
- Form token detection
- On-demand revalidation
- Slug generation

### 8.3 Later

- Full Google Drive webhook support

## 9. Operational Workflow

### 9.1 Authoring

1. The operator creates a Google Doc in the target Drive folder.
2. The operator writes content using normal Google Docs editing features.
3. The system detects the document and registers it in the management spreadsheet.

### 9.2 Publication

1. The operator opens the spreadsheet.
2. The operator sets `status` to `published`.
3. The operator optionally sets `publishedAt`, `theme`, `accentColor`, and `layout`.
4. The system syncs the content.
5. The Next.js site rebuilds or revalidates the affected route.

### 9.3 Update

1. The operator edits the Google Doc.
2. The backend detects the updated timestamp.
3. The affected page is revalidated.
4. Visitors see the updated static page.

## 10. Design Rules

The public website should not inherit visual styles directly from Google Docs.

Rules:

- Strip inline styles from imported HTML.
- Preserve semantic structure.
- Apply all visual design through Tailwind and React components.
- Keep themes controlled through whitelisted class names.
- Apply accent colors through validated CSS variables.
- Sanitize all embedded HTML.
- Convert special tokens such as forms into controlled components.

This prevents content editors from accidentally breaking the site design.

## 11. Security and Permissions

Initial design assumes a service account or OAuth-based integration with limited Google Drive access.

Security requirements:

- Limit access to configured Drive folders.
- Store Google credentials securely.
- Do not expose Google API tokens to the frontend.
- Sanitize all document-derived HTML.
- Validate theme class names against an allowlist.
- Validate color values before using them as CSS variables.
- Restrict iframe embeds to approved Google Forms URLs.
- Keep unpublished content out of public frontend responses.

## 12. Key Risks

### 12.1 Google Docs HTML Quality

Google Docs export HTML may include unnecessary markup. The parser must normalize the output and remove document-specific styling.

### 12.2 Image Handling

Images embedded in Google Docs may require special handling for extraction, hosting, permissions, and optimization.

### 12.3 Webhook Complexity

Google Drive change detection can be more complex than simple polling. The MVP may begin with scheduled sync or manual sync before implementing full webhook support.

### 12.4 User Expectations

Users may expect Google Docs layout to match the website exactly. The product must explain that Google Docs controls content structure, while the website controls design.

## 14. Recommended Implementation Order

The project should keep the full architecture in view while building from the smallest useful publishing loop.

### Phase 0: Repository and Public-Safe Foundation

- Initialize the repository.
- Add `README.md`, `.gitignore`, `.env.example`, and license files.
- Keep real Google credentials, private folder IDs, and customer data out of Git.
- Define environment variables before implementing API calls.
- Add a minimal local development command.
- Add Docker and Docker Compose files before the first runnable MVP.

Goal:

```text
The repository can be public from day one without leaking secrets.
```

### Phase 1: Manual Data Pipeline

- Create a sample Google Sheet manually.
- Create one sample Google Doc manually.
- Read the sheet through the Google Sheets API.
- Fetch the linked Google Doc.
- Convert the document to clean HTML.
- Return normalized post JSON from a local API function.

Goal:

```text
Google Docs + Google Sheets can produce one normalized article object.
```

### Phase 2: Next.js Rendering MVP

- Create the Next.js frontend.
- Render a post list from the normalized content.
- Render a post detail page by slug.
- Add published/draft filtering.
- Add `publishedAt` filtering.
- Add basic sanitization before rendering HTML.

Goal:

```text
Changing a sheet row to published makes the article appear on the site.
```

### Phase 3: Fixed Pages

- Add `/Pages` folder support.
- Map page file names to public routes.
- Support a root page such as `top`, `home`, or `index`.
- Apply the same clean HTML pipeline used for posts.

Goal:

```text
Google Docs can manage both posts and fixed pages.
```

### Phase 4: Design Controls

- Add a small allowlist of themes.
- Add a small allowlist of layout variants.
- Add validated accent color support.
- Apply the selected values through React components and CSS variables.

Initial theme examples:

- `default`
- `notice`
- `simple`

Goal:

```text
Sheets can control presentation safely without letting editors break the site.
```

### Phase 5: Sync and Revalidation

- Add a manual sync endpoint.
- Add cache invalidation.
- Add on-demand revalidation for changed pages.
- Add basic logging for failed syncs.
- Keep Google Drive webhook support for a later phase unless polling becomes too slow.

Goal:

```text
Operators can update Google Docs and refresh the public site without a redeploy.
```

### Phase 6: Setup Automation

- Detect new documents in managed folders.
- Auto-register documents in the management spreadsheet.
- Auto-create a management spreadsheet when missing.
- Add dropdown validation to generated spreadsheets.

Goal:

```text
New content can enter the CMS workflow with minimal manual setup.
```

### Phase 7: Production Hardening

- Add tests for HTML cleaning and metadata validation.
- Add rate limit handling for Google APIs.
- Add error reporting.
- Add deployment documentation.
- Add backup and recovery notes.
- Add permission setup documentation for Google Drive.

Goal:

```text
The system is stable enough to run for a real small organization.
```

## 15. Public Git Repository Policy

The project can be managed as a public Git repository, but the repository must be designed as public-safe from the beginning.

### 15.1 What Can Be Public

- Application source code
- Example schemas
- Example Google Sheet column definitions
- Example HTML output
- Local development Docker files
- VPS Docker Compose files without secrets
- Documentation
- Demo content with fake data

### 15.2 What Must Not Be Public

- Service account JSON files
- OAuth client secrets
- Real Google Drive folder IDs for customers
- Real spreadsheet IDs for customers
- Real document IDs for private content
- API tokens
- Customer content exports
- Logs containing Google API responses

### 15.3 Required Files

The repository should include:

- `.gitignore`
- `.env.example`
- `README.md`
- `LICENSE`
- `docs/setup-google-api.md`
- `docs/content-model.md`

The `.env.example` file should define variable names without real values.

Example:

```text
GOOGLE_CLIENT_EMAIL=
GOOGLE_PRIVATE_KEY=
GOOGLE_PROJECT_ID=
GOOGLE_DRIVE_ROOT_FOLDER_ID=
GOOGLE_POSTS_SHEET_ID=
NEXT_PUBLIC_SITE_URL=
REVALIDATE_SECRET=
```

### 15.4 Recommended Repository Strategy

Use one public repository for the core framework.

Keep customer-specific configuration outside the repository through environment variables or private deployment settings.

Recommended structure:

```text
gdcms/
- apps/
  - web/
  - api/
- packages/
  - google-content/
  - html-cleaner/
  - schemas/
- docs/
- examples/
```

For the first MVP, a simpler single Next.js app is acceptable:

```text
gdcms/
- src/
- docs/
- examples/
- PLAN.md
```

The monorepo structure can be introduced later if the backend and frontend need to be deployed separately.

### 15.5 License Direction

The initial license should be chosen intentionally.

Options:

- MIT: easiest for adoption, permissive.
- Apache-2.0: permissive with explicit patent terms.
- AGPL-3.0: forces hosted derivatives to share source, but may reduce adoption.

Initial recommendation:

```text
Use MIT or Apache-2.0 while the goal is adoption and experimentation.
```

If the project later becomes a commercial hosted product, keep the public repository as the open core and run customer-specific deployment/configuration privately.

## 16. Product Positioning

This CMS is positioned as:

> A website CMS operated entirely from Google Drive.

The main value is not replacing Google Docs or Google Sheets. The value is turning them into a structured publishing workflow for modern static websites.

Target users:

- Small businesses
- Local organizations
- Clinics
- Schools and classrooms
- Welfare facilities
- Community groups
- NPOs
- Local event organizers

The system is especially useful for teams that already use Google Drive but do not want to learn WordPress or a custom CMS admin screen.
